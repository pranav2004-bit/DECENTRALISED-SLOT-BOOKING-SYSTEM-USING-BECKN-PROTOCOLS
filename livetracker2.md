# BECKN Platform — Business Workflow & Inventory Live Tracker

> **Project scope declaration (2026-07-17), inherited from [livetracker1.md](livetracker1.md):** this project builds a **private, self-contained Beckn network** — our own Registry, Gateway, BAP, and BPP — that strictly follows the Beckn Protocol Specification and Beckn-ONDC Implementation Guidelines for correctness and interop-readiness. It does **not** connect to, register with, or integrate with the real, live ONDC network. Everything in this tracker is business-workflow engineering on top of the already-completed trust layer, not preparation for a live ONDC handoff. `project_details.md` is never modified — it is the frozen, client-provided brief.

**Scope of this tracker:** Real-time inventory management and the core Beckn business transaction workflow (`search → select → init → confirm`, plus `status`/`cancel`/`update`/`track`/`rating`/`support`) across BAP, Gateway, and BPP, built on top of the trust layer `livetracker1.md` closed — including the security, performance, observability, and UI/UX work needed to call this layer genuinely production-ready, not just functionally complete. Starts with **one category (Beauty)** proven end-to-end, then widens to Healthcare and Automotive. **Payment gateway integration is explicitly out of scope for this tracker** — tracked separately as a future `livetracker3.md`, once booking itself is solid.

Key confirmed facts baked into this tracker:
- `Order.status` is a fixed 3-value enum: `ACTIVE`, `COMPLETE`, `CANCELLED`.
- `Fulfillment.state` has **no fixed enum in the core protocol** — it's a free-text, network-policy-defined code. This project defines its own fulfillment-state codes (§1.3 below); it does not assume ONDC's unpublished values.
- `Cancellation.cancelled_by` is `CONSUMER` or `PROVIDER` only — no `SYSTEM` value exists in the real schema.
- Exact ONDC domain codes for Healthcare/Automotive **service** booking remain genuinely unconfirmed after real research (only retail-of-goods codes like `RET13`/`RET18`/`RET19` and the `ONDC:SRV` services-vertical prefix are publicly confirmed) — not a blocker, since this private network defines its own domain codes (§4.1/§4.2), using ONDC's naming pattern as a reference only, consistent with `livetracker1.md`'s scope declaration.

## How to Use This Tracker

1. Work phases **top to bottom**. Phase 1 (generic inventory core) must complete before Phase 2, which must complete before Phase 3 — each phase's minimal deliverable is a real prerequisite for the next, not an arbitrary ordering.
2. A task/subtask is checked `[x]` **only** after its implementation is complete **and** its Testing & Validation Gate passes. Never check a box on implementation alone.
3. A phase is closed **only** after its "Phase Exit — Testing & Sign-off" section is fully checked.
4. **Vertical-slice discipline:** Phase 3 wires BAP, Gateway, and BPP together for **one category (Beauty) only**. Do not build Healthcare/Automotive support before Phase 3's Beauty slice is proven live end-to-end — that is the walking-skeleton principle this tracker is built around, validated by how `livetracker1.md`'s own Phase 1 (all four apps together) and Phase 4.1 (live trust-chain verification) already worked.
5. If implementation reveals a decision not captured in a component's `*_details.md`, update that file in the same work session with an additive "Implementation note" callout — never rewrite the original client-provided content, and never touch `project_details.md`. Also update whichever root-level doc (`SECURITY.md`, `API_CONVENTIONS.md`, `OBSERVABILITY.md`, `TESTING.md`) the change actually affects — see the doc-extension list above.
6. Lifecycle tags on each item show when it's required. Do not build ahead of the tagged stage — that's over-engineering. Do not skip a tag at its stage — that's under-engineering. Every task below carries an explicit tag; absence of a tag is never used as an excuse to skip it.
7. When a shared module changes after other phases already depend on it (e.g. a Phase 1 state-machine fix after Phase 3 is built), re-run every downstream phase's affected test gates before considering the change safe — the same "prove the whole chain again, not just the new part" discipline already used throughout `livetracker1.md`.

**Lifecycle tags:** `[MVP]` build now · `[PILOT]` small real-batch of participants · `[BETA]` broader external participants · `[ENT]` enterprise scale, design-documented now but not built until real scale demands it (avoids over-engineering).

**Test type legend:** `SMOKE` basic runs · `SANITY` narrow correctness check after a change · `FUNC` functional correctness · `POS`/`NEG`/`EDGE` case coverage · `INTEG` cross-service · `E2E` full flow · `REGR` regression · `SEC` security/abuse · `LOAD` concurrency/throughput/performance · `DR` failure-injection/disaster-recovery.

## Phase 1 — Generic Inventory & Booking Core (Shared Library)

**Objective:** Build a domain-agnostic Resource/Slot/Availability/Capacity/Booking data model as a shared library (`shared/inventory_core/`, same pattern as `shared/beckn_crypto/` and `shared/event_bus/`) — built once, usable by Beauty now and Healthcare/Automotive later, before any business-workflow wiring begins.

### 1.1 Core Data Model — Resource, Slot, Availability Calendar `[MVP]`
- [x] `Resource` entity — the bookable person/thing (doctor, stylist, bay+mechanic), fields aligned to the confirmed real `Item`/`Provider` schema shapes, not invented independently
- [x] `Slot` entity — a bookable time window on a resource (`start_time`, `end_time`, `capacity_remaining`, `status`)
- [x] `Availability Calendar` — recurring schedule + exceptions/holidays, generates real `Slot` rows; field shape follows the confirmed real `Time`/`Schedule` schemas (`frequency`, `holidays`, `times`, `range`)
- [x] Database indexes on the actual high-frequency query paths (resource+time-range lookups, status filters) — decided from the real query patterns Phase 3 exercises, not guessed in advance
- **Test Gate:** `SMOKE`/`FUNC`/`EDGE` — module imports and migrates cleanly; slot generation from a recurring rule plus at least one holiday exception produces the correct slot set; a malformed recurrence rule is rejected, not silently miscalculated. **Passed 2026-07-18** — `shared/inventory_core` built as a real Django app (`Resource`/`Slot`/`AvailabilityCalendar` in `shared/inventory_core/models.py`), installed into BPP (its first consumer, `INSTALLED_APPS`), migrated cleanly against real Postgres (`bpp-db`), verified importable via real `manage.py shell`. 12 real tests in `BPP/backend/core/test_inventory_core.py` (run there, not `shared/inventory_core/tests.py`, since it's a Django app needing a real settings module + DB — the existing project pattern for exercising `shared/` libs from a consuming app's own suite): recurring-rule + holiday-exclusion slot generation, weekday filtering, 7 malformed-rule EDGE cases (zero/negative frequency, zero duration, inverted range, malformed time/weekday/holiday tokens) all rejected via `ValidationError` with zero partial slots created, plus the `Slot` `CheckConstraint`s enforced at the real DB level. Full BPP suite (34 tests) re-run clean, `ruff check` clean. One real bug caught and fixed during this gate: `generate_slots()` originally filtered each slot's own timestamp against `range_end`, which silently dropped a valid final day's slots whenever `range_end`'s time-of-day fell before that day's `times` anchors — fixed to bound by calendar day instead (see `shared/inventory_core/models.py` `generate_slots()` docstring/comment). Protocol grounding for `Time`/`Schedule`/`Descriptor` (not previously pulled) recorded in `protocol_compliance_notes_v1.1.md` §F before implementation.

### 1.2 Capacity & Concurrency Safety `[MVP]`
- [x] Atomic, race-safe capacity decrement at the database level (not check-then-write)
- [x] Short-lived technical lock during slot mutation, distinct from the business-level reservation in §1.3 — per the inventory-patterns doc's Reservation-vs-Lock distinction, itself validated against the real protocol's `Order`/`Fulfillment` split
- **Test Gate:** `LOAD`/`NEG` — genuine concurrent-write test: N parallel confirm attempts against a capacity-1 slot must yield exactly one success and N-1 clean rejections, not a corrupted count. Not a theoretical claim — must be run against real Postgres, not mocked. **Passed 2026-07-18** — `SlotManager.try_reserve()` (single conditional `UPDATE ... WHERE capacity_remaining >= quantity`, `F()`-expression decrement, no check-then-write) and `SlotManager.lock_for_mutation()` (`select_for_update()` context manager, distinct from §1.3's Redis-backed `HELD` reservation, only usable inside `transaction.atomic()`) added to `shared/inventory_core/models.py`. 10 real concurrent `try_reserve` attempts against a real capacity-1 `Slot` in real Postgres (`BPP/backend/core/test_inventory_core_concurrency.py`, `pytest.mark.django_db(transaction=True)` + `ThreadPoolExecutor`, one real DB connection per thread) yielded exactly 1 success / 9 clean rejections, re-run 5x clean, no corrupted count. Added a `capacity_remaining >= 0` `CheckConstraint` as DB-level defense-in-depth (confirmed via `makemigrations` that Django's `PositiveIntegerField` does *not* already add this at the DB level — a real gap, not a redundant addition). Full BPP suite (42 tests) and `ruff` re-run clean.

### 1.3 Booking & Fulfillment State Machines (two separate machines, not one) `[MVP]`
- [x] **Booking state machine** — mirrors the confirmed real `Order.status` enum exactly (`ACTIVE` / `COMPLETE` / `CANCELLED`), plus a pre-commit `HELD` state for the reservation window before an order becomes `ACTIVE`
- [x] **Fulfillment state machine** — this project's own network-policy-defined codes (since the real protocol leaves `Fulfillment.state` free-text): `SCHEDULED` → `IN_PROGRESS` → `COMPLETED`, with `NO_SHOW` as an alternate terminal state. Documented explicitly as project-defined, not claimed as an ONDC-confirmed value.
- [x] Reservation Window / TTL-based `HELD` state, Redis-backed — reuses the real Redis dependency and TTL/state pattern already proven live via Phase 4.2's `RedisCircuitBreaker` (`shared/resilient_http/circuit_breaker.py`). **Correction, precise:** earlier drafts of this line cited "Phase 2's rate limiter" as a second Redis precedent — that's factually wrong. `RUNBOOK.md`/`DATABASE.md` both document Registry's rate limiter as still in-memory/per-worker, an accepted `[MVP]` limitation explicitly deferred to `[BETA]`, not a Redis pattern to copy. The one real, live-proven Redis precedent here is the circuit breaker.
- **Test Gate:** `FUNC`/`EDGE` — every valid transition succeeds, every invalid transition (e.g. `CANCELLED` → `ACTIVE`) is rejected; a `HELD` slot with an expired TTL auto-returns to `AVAILABLE` without manual intervention, verified live against real Redis. **Passed 2026-07-18** — added `Booking` (two independent `status`/`fulfillment_status` fields, each with its own explicit valid-transition graph enforced by `transition_status()`/`transition_fulfillment_status()`, raising `ValidationError` on any other edge) and `shared/inventory_core/reservation.py` (`ReservationHold` Redis TTL primitive + `hold_slot()`/`release_expired_hold()`/`confirm_hold()` orchestration, combining `Slot.objects.lock_for_mutation()` from §1.2 with a real Redis client — kept out of `models.py` as a separate service-layer module, matching BPP's own `core/onboarding_service.py` pattern, since it coordinates Postgres + Redis together). 21 real tests in `BPP/backend/core/test_inventory_core_booking.py`: all 4 valid + 4 invalid Booking-status edges, all 3 valid + 4 invalid Fulfillment-status edges, and the TTL scenario itself — a real 1-second Redis key genuinely expires (`time.sleep`, no mocking), then `release_expired_hold()` flips the `Slot` back to `AVAILABLE` with capacity restored and the `Booking` to `CANCELLED`, with zero manual intervention. TTL-timing tests re-run 3x to rule out flakiness before trusting them. Expiry detection is deliberately opportunistic (checked on next touch, not a background keyspace-notification listener) — right-sized for `[MVP]`, since no task-queue/worker process exists yet in any of the four apps; documented as a deliberate choice, not an oversight. Full BPP suite (63 tests) and `ruff` re-run clean.

### 1.4 Event Schema, Versioning & Wiring to `shared/event_bus` `[MVP]`
- [x] Adopt a confirmed, real-protocol-aligned event vocabulary: `SlotCreated`, `SlotPublished`, `SlotReserved`, `SlotLocked`, `SlotConfirmed`, `SlotReleased`, `SlotCancelled`, `SlotCompleted`, `BookingConfirmed`, `BookingCancelled`
- [x] Every event carries a `version` field in its metadata from day one — cheap to add now, expensive to retrofit once Healthcare/Automotive consumers exist in Phase 4; a version bump must never break an existing consumer without a documented migration
- [x] Idempotent event handlers — every consumer checks a unique event ID before acting, since At-Least-Once delivery (the only realistic guarantee) means duplicate delivery *will* happen
- [x] Wire into the existing `shared/event_bus` (Redis-backed, DLQ, built in Phase 1 of `livetracker1.md`) — reused, not rebuilt
- [x] **Per-entity event ordering, not just delivery.** Idempotency (above) handles duplicates; it does not by itself guarantee `SlotReserved` is processed before `SlotConfirmed` for the *same* slot if both arrive close together. Events for the same `Slot`/`Booking` ID must be processed in the order they occurred — global ordering across all slots is not required, only per-entity ordering, per the EDA-principles research.
- [x] **Scope boundary, explicit:** this event bus carries **internal**, cross-module coordination within one app (e.g. BPP's Inventory module telling its own Fulfillment module a slot was confirmed) — matching BAP's and BPP's own documented architecture ("Modular Monolith" with an internal Event-Driven Architecture layer, per both `*_details_v1.1.md`'s "Internal Processing Architecture"/"Internal Communication Mechanism" sections). It never substitutes for the **external** Beckn protocol calls between BAP ↔ Gateway ↔ BPP (`search`/`select`/`init`/`confirm`/...), which stay strictly signed HTTP request/callback per the trust layer — the two are related but never conflated.
- **Test Gate:** `FUNC`/`NEG` — the exact same event delivered twice produces one business effect, not two; a deliberately poisoned event lands in the DLQ instead of retrying forever; a consumer given an unexpected event version fails safely (logs and skips) instead of crashing. **Passed 2026-07-19** — added `shared/inventory_core/events.py` (the `SlotEvent`/`BookingEvent` vocabulary, `publish_event()` always embedding a `version` field, `process_event()` idempotency wrapper meant to be composed with `shared/event_bus`'s own already-proven `process_with_dlq` — that function itself untouched, reused exactly as documented) and a `ProcessedEvent` idempotency-ledger model (kept in `models.py`, not `events.py`, since Django's migration autodiscovery only scans an app's `models.py` — a real gap caught while implementing, not a style choice). Wired real emission into every trigger point Phase 1.1–1.3 already has (`generate_slots`→`SlotCreated`, `hold_slot`→`SlotReserved`, `confirm_hold`→`SlotConfirmed`+`BookingConfirmed`, `release_expired_hold`→`SlotReleased`+`BookingCancelled`) via an optional `event_bus=` kwarg, `None` by default, so every existing 1.1–1.3 call site keeps working unchanged. `SlotPublished`/`SlotLocked`/`SlotCancelled`/`SlotCompleted` are defined in the vocabulary but have no trigger point yet — documented explicitly as deferred to Phase 2/3's real business flows, not silently unaddressed. Per-entity ordering demonstrated (not just asserted) against the real bus: two events for the same slot, consumed in publish order. 9 new tests plus the full 72-test BPP suite (re-run twice) and `ruff` all clean.

### 1.5 Domain Adapter Interface (extension point only — no domain built yet) `[MVP]`
- [x] A thin adapter interface so domain-specific fields (consultation type, combo services, multi-resource requirements) plug into the generic core without forking it — validated against the real `Fulfillment.type` field being explicitly domain-policy-defined
- **Test Gate:** `FUNC` — a stub/mock domain adapter exercises the interface end-to-end; the real Beauty adapter itself is built in Phase 2, not here. **Passed 2026-07-19** — `shared/inventory_core/domain_adapter.py`'s abstract `DomainAdapter` (`validate_resource_domain_data`/`validate_booking_domain_data`/`required_resource_count`/`fulfillment_type`) plus a `register_adapter`/`get_adapter` registry (raises `LookupError` for an unregistered domain, never silently `None`). A `StubDomainAdapter` (test-only, in `BPP/backend/core/test_inventory_core_domain_adapter.py`, not shipped in `shared/`) exercises validation, the multi-resource hook (a stub "combo" requiring 2 resources vs. 1 for standard), and `fulfillment_type` end-to-end; confirmed the ABC itself can't be instantiated directly. 7 new tests, full 79-test BPP suite and `ruff` clean.
