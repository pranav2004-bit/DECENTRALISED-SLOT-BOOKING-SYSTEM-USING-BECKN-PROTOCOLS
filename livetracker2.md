# BECKN Platform — Business Workflow & Inventory Live Tracker

> **Project scope declaration (2026-07-17), inherited from [livetracker1.md](livetracker1.md):** this project builds a **private, self-contained Beckn network** — our own Registry, Gateway, BAP, and BPP — that strictly follows the Beckn Protocol Specification and Beckn-ONDC Implementation Guidelines for correctness and interop-readiness. It does **not** connect to, register with, or integrate with the real, live ONDC network. Everything in this tracker is business-workflow engineering on top of the already-completed trust layer, not preparation for a live ONDC handoff. `project_details.md` is never modified — it is the frozen, client-provided brief.

**Scope of this tracker:** Real-time inventory management and the core Beckn business transaction workflow (`search → select → init → confirm`, plus `status`/`cancel`/`update`/`track`/`rating`/`support`) across BAP, Gateway, and BPP, built on top of the trust layer `livetracker1.md` closed — including the security, performance, observability, and UI/UX work needed to call this layer genuinely production-ready, not just functionally complete. Starts with **one category (Beauty)** proven end-to-end, then widens to Healthcare and Automotive. **Payment gateway integration is explicitly out of scope for this tracker** — tracked separately as a future `livetracker3.md`, once booking itself is solid.

**Protocol grounding:** Before drafting this tracker, the actual core transaction schema was pulled directly from the official [beckn/protocol-specifications](https://github.com/beckn/protocol-specifications) repository (`schema/Order.yaml`, `Item.yaml`, `Fulfillment.yaml`, `FulfillmentState.yaml`, `Catalog.yaml`, `Provider.yaml`, `Quotation.yaml`, `Cancellation.yaml`, `Category.yaml`) — real, confirmed field names, not inferred from the reference PDFs. Key confirmed facts baked into this tracker:
- `Order.status` is a fixed 3-value enum: `ACTIVE`, `COMPLETE`, `CANCELLED`.
- `Fulfillment.state` has **no fixed enum in the core protocol** — it's a free-text, network-policy-defined code. This project defines its own fulfillment-state codes (§1.3 below); it does not assume ONDC's unpublished values.
- `Cancellation.cancelled_by` is `CONSUMER` or `PROVIDER` only — no `SYSTEM` value exists in the real schema.
- Exact ONDC domain codes for Healthcare/Automotive **service** booking remain genuinely unconfirmed after real research (only retail-of-goods codes like `RET13`/`RET18`/`RET19` and the `ONDC:SRV` services-vertical prefix are publicly confirmed) — not a blocker, since this private network defines its own domain codes (§4.1/§4.2), using ONDC's naming pattern as a reference only, consistent with `livetracker1.md`'s scope declaration.

**Docs this tracker will extend as it proceeds** (in addition to the `*_details_v1.1.md` Implementation-note convention already established): `SECURITY.md` gains a Business Layer Security section (§3.7 below); `API_CONVENTIONS.md` gains the new business endpoint conventions; `OBSERVABILITY.md` gains the business-metrics extension (§3.10); `TESTING.md` gains the `SEC`/`LOAD` abuse-simulation and race-condition test patterns introduced here; `protocol_compliance_notes_v1.1.md` gains a new section recording the confirmed `Order`/`Item`/`Fulfillment` schema facts above. ## How to Use This Tracker

1. Work phases **top to bottom**. Phase 1 (generic inventory core) must complete before Phase 2, which must complete before Phase 3 — each phase's minimal deliverable is a real prerequisite for the next, not an arbitrary ordering.
2. A task/subtask is checked `[x]` **only** after its implementation is complete **and** its Testing & Validation Gate passes. Never check a box on implementation alone.
3. A phase is closed **only** after its "Phase Exit — Testing & Sign-off" section is fully checked.
4. **Vertical-slice discipline:** Phase 3 wires BAP, Gateway, and BPP together for **one category (Beauty) only**. Do not build Healthcare/Automotive support before Phase 3's Beauty slice is proven live end-to-end — that is the walking-skeleton principle this tracker is built around, validated by how `livetracker1.md`'s own Phase 1 (all four apps together) and Phase 4.1 (live trust-chain verification) already worked.
5. If implementation reveals a decision not captured in a component's `*_details.md`, update that file in the same work session with an additive "Implementation note" callout — never rewrite the original client-provided content, and never touch `project_details.md`. Also update whichever root-level doc (`SECURITY.md`, `API_CONVENTIONS.md`, `OBSERVABILITY.md`, `TESTING.md`) the change actually affects — see the doc-extension list above.
6. Lifecycle tags on each item show when it's required. Do not build ahead of the tagged stage — that's over-engineering. Do not skip a tag at its stage — that's under-engineering. Every task below carries an explicit tag; absence of a tag is never used as an excuse to skip it.
7. When a shared module changes after other phases already depend on it (e.g. a Phase 1 state-machine fix after Phase 3 is built), re-run every downstream phase's affected test gates before considering the change safe — the same "prove the whole chain again, not just the new part" discipline already used throughout `livetracker1.md`.

**Lifecycle tags:** `[MVP]` build now · `[PILOT]` small real-batch of participants · `[BETA]` broader external participants · `[ENT]` enterprise scale, design-documented now but not built until real scale demands it (avoids over-engineering).

**Test type legend:** `SMOKE` basic runs · `SANITY` narrow correctness check after a change · `FUNC` functional correctness · `POS`/`NEG`/`EDGE` case coverage · `INTEG` cross-service · `E2E` full flow · `REGR` regression · `SEC` security/abuse · `LOAD` concurrency/throughput/performance · `DR` failure-injection/disaster-recovery.

## Phase 1 — Generic Inventory & Booking Core (Shared Library)

**Objective:** Build a domain-agnostic Resource/Slot/Availability/Capacity/Booking data model as a shared library (`shared/inventory_core/`, same pattern as `shared/beckn_crypto/` and `shared/event_bus/`) — built once, usable by Beauty now and Healthcare/Automotive later, before any business-workflow wiring begins.

### 1.1 Core Data Model — Resource, Slot, Availability Calendar `[MVP]`
- [ ] `Resource` entity — the bookable person/thing (doctor, stylist, bay+mechanic), fields aligned to the confirmed real `Item`/`Provider` schema shapes, not invented independently
- [ ] `Slot` entity — a bookable time window on a resource (`start_time`, `end_time`, `capacity_remaining`, `status`)
- [ ] `Availability Calendar` — recurring schedule + exceptions/holidays, generates real `Slot` rows; field shape follows the confirmed real `Time`/`Schedule` schemas (`frequency`, `holidays`, `times`, `range`)
- [ ] Database indexes on the actual high-frequency query paths (resource+time-range lookups, status filters) — decided from the real query patterns Phase 3 exercises, not guessed in advance
- **Test Gate:** `SMOKE`/`FUNC`/`EDGE` — module imports and migrates cleanly; slot generation from a recurring rule plus at least one holiday exception produces the correct slot set; a malformed recurrence rule is rejected, not silently miscalculated.

### 1.2 Capacity & Concurrency Safety `[MVP]`
- [ ] Atomic, race-safe capacity decrement at the database level (not check-then-write)
- [ ] Short-lived technical lock during slot mutation, distinct from the business-level reservation in §1.3 — per the inventory-patterns doc's Reservation-vs-Lock distinction, itself validated against the real protocol's `Order`/`Fulfillment` split
- **Test Gate:** `LOAD`/`NEG` — genuine concurrent-write test: N parallel confirm attempts against a capacity-1 slot must yield exactly one success and N-1 clean rejections, not a corrupted count. Not a theoretical claim — must be run against real Postgres, not mocked.

### 1.3 Booking & Fulfillment State Machines (two separate machines, not one) `[MVP]`
- [ ] **Booking state machine** — mirrors the confirmed real `Order.status` enum exactly (`ACTIVE` / `COMPLETE` / `CANCELLED`), plus a pre-commit `HELD` state for the reservation window before an order becomes `ACTIVE`
- [ ] **Fulfillment state machine** — this project's own network-policy-defined codes (since the real protocol leaves `Fulfillment.state` free-text): `SCHEDULED` → `IN_PROGRESS` → `COMPLETED`, with `NO_SHOW` as an alternate terminal state. Documented explicitly as project-defined, not claimed as an ONDC-confirmed value.
- [ ] Reservation Window / TTL-based `HELD` state, Redis-backed — reuses the real Redis dependency and TTL/state pattern already proven live via Phase 4.2's `RedisCircuitBreaker` (`shared/resilient_http/circuit_breaker.py`). **Correction, precise:** earlier drafts of this line cited "Phase 2's rate limiter" as a second Redis precedent — that's factually wrong. `RUNBOOK.md`/`DATABASE.md` both document Registry's rate limiter as still in-memory/per-worker, an accepted `[MVP]` limitation explicitly deferred to `[BETA]`, not a Redis pattern to copy. The one real, live-proven Redis precedent here is the circuit breaker.
- **Test Gate:** `FUNC`/`EDGE` — every valid transition succeeds, every invalid transition (e.g. `CANCELLED` → `ACTIVE`) is rejected; a `HELD` slot with an expired TTL auto-returns to `AVAILABLE` without manual intervention, verified live against real Redis.

### 1.4 Event Schema, Versioning & Wiring to `shared/event_bus` `[MVP]`
- [ ] Adopt a confirmed, real-protocol-aligned event vocabulary: `SlotCreated`, `SlotPublished`, `SlotReserved`, `SlotLocked`, `SlotConfirmed`, `SlotReleased`, `SlotCancelled`, `SlotCompleted`, `BookingConfirmed`, `BookingCancelled`
- [ ] Every event carries a `version` field in its metadata from day one — cheap to add now, expensive to retrofit once Healthcare/Automotive consumers exist in Phase 4; a version bump must never break an existing consumer without a documented migration
- [ ] Idempotent event handlers — every consumer checks a unique event ID before acting, since At-Least-Once delivery (the only realistic guarantee) means duplicate delivery *will* happen
- [ ] Wire into the existing `shared/event_bus` (Redis-backed, DLQ, built in Phase 1 of `livetracker1.md`) — reused, not rebuilt
- [ ] **Per-entity event ordering, not just delivery.** Idempotency (above) handles duplicates; it does not by itself guarantee `SlotReserved` is processed before `SlotConfirmed` for the *same* slot if both arrive close together. Events for the same `Slot`/`Booking` ID must be processed in the order they occurred — global ordering across all slots is not required, only per-entity ordering, per the EDA-principles research.
- [ ] **Scope boundary, explicit:** this event bus carries **internal**, cross-module coordination within one app (e.g. BPP's Inventory module telling its own Fulfillment module a slot was confirmed) — matching BAP's and BPP's own documented architecture ("Modular Monolith" with an internal Event-Driven Architecture layer, per both `*_details_v1.1.md`'s "Internal Processing Architecture"/"Internal Communication Mechanism" sections). It never substitutes for the **external** Beckn protocol calls between BAP ↔ Gateway ↔ BPP (`search`/`select`/`init`/`confirm`/...), which stay strictly signed HTTP request/callback per the trust layer — the two are related but never conflated.
- **Test Gate:** `FUNC`/`NEG` — the exact same event delivered twice produces one business effect, not two; a deliberately poisoned event lands in the DLQ instead of retrying forever; a consumer given an unexpected event version fails safely (logs and skips) instead of crashing.

### 1.5 Domain Adapter Interface (extension point only — no domain built yet) `[MVP]`
- [ ] A thin adapter interface so domain-specific fields (consultation type, combo services, multi-resource requirements) plug into the generic core without forking it — validated against the real `Fulfillment.type` field being explicitly domain-policy-defined
- **Test Gate:** `FUNC` — a stub/mock domain adapter exercises the interface end-to-end; the real Beauty adapter itself is built in Phase 2, not here.

### Phase 1 Exit — Testing & Sign-off
- [ ] All Phase 1 test gates above pass, none silently skipped
- [ ] `shared/inventory_core` importable and used identically by BPP's Django app (proven via real `manage.py shell` import, not a bare `python -c` check — see the known Phase-1-of-`livetracker1.md` methodology gotcha)
- [ ] `DATABASE.md`'s backup/restore dry-run re-run specifically against BPP's database with the new Resource/Slot/Booking tables present — the existing procedure is documented as proven for Registry and stated to "apply" to BAP/BPP, but this is the first time BPP's schema actually changes meaningfully since that claim was written; re-verified, not just assumed to still hold
- [ ] `ARCHITECTURE.md` updated to list `shared/inventory_core` alongside the other shared libraries
- [ ] Sign-off recorded

"Minimal" means the smallest **permanent, production-grade** slice — not a throwaway prototype to be rewritten later. Not the full-featured profile/inventory management suite.

- [ ] Customer sessions stored in Redis (`django-redis`, already a real dependency in this codebase) rather than Django's default DB-backed session store — directly matches `project_details.md`'s explicit "Redis for caching and session management" prerequisite, not just the caching half of it

### 2.2 BPP — Business Account & Beauty Domain Adapter `[MVP]`
- [ ] One business-account login (salon owner/admin) — minimal fields; **not** individual staff logins (deferred to Phase 4, tagged `[PILOT]` there)

- [ ] Minimal **Provider Lifecycle Management** (business account `ACTIVE`/`INACTIVE` state) — named explicitly in [BPP_details_v1.1.md](BPP/BPP_details_v1.1.md)'s Provider Management Module breakdown, same reasoning as §2.1. **Provider Configuration Management** (the module's other named peer) is satisfied by the Availability Calendar itself (§1.1/§2.2) — a business's operating-hours/schedule configuration *is* its configuration management here, not a separate settings screen — documented explicitly as the mapping, not silently left unaddressed.
- **Test Gate:** `E2E` — the business account creates at least one real `Resource` and generates real `Slot` rows via its Availability Calendar, confirmed by direct DB inspection, not just a 200 response; a deactivated business account's inventory stops appearing in search.

### 2.3 Minimal Catalog Exposure `[MVP]`
- [ ] BPP's Beauty catalog represented internally using the confirmed real `Catalog`/`Provider`/`Item` schema shapes — not yet wired to `search`/`on_search` (that's Phase 3)

### 2.4 Minimal Web UI Foundation `[MVP]`
- [ ] A single, shared, **mobile-first** responsive layout shell for both BAP web and BPP web (Next.js/Tailwind, the already-confirmed stack) — designed at the smallest viewport first, then progressively enhanced for tablet/desktop, not the reverse

- [ ] Base component library: loading state, empty state, error state, form-input pattern — built once, reused across every screen Phase 3 adds, so booking-flow screens don't each reinvent feedback patterns
- [ ] Baseline accessibility: semantic HTML, sufficient color contrast, keyboard navigable forms — not a full WCAG audit yet, but not ignored either
- **Test Gate:** `FUNC` — the shell renders correctly at mobile (375px), tablet, and desktop breakpoints with no layout breakage; keyboard-only navigation reaches every interactive element in the shell; a real WebSocket connection between web app and backend opens and stays alive, confirmed by inspecting actual socket traffic.

## Phase 3 — Beauty Booking Workflow, End-to-End

**Objective:** Wire the real Beckn transaction flow across BAP ↔ Gateway ↔ BPP for Beauty, verified live over the real Docker network — not mocked, not simulated — and harden it to a genuinely production-ready standard before widening to other domains.

### 3.1 Discovery — `search` / `on_search` `[MVP]`
- [ ] Extend Gateway's existing `POST /search` plumbing (built as trust-chain-only in Phase 4.1 of `livetracker1.md`) with real business logic: route to `SUBSCRIBED` Beauty BPPs, return real catalog data
- [ ] BAP-side search request builder + `on_search` response handler
- **Test Gate:** `E2E` — a real search for a Beauty service returns a real salon and real service listing over the live Docker network, signed and verified per the existing trust layer.

### 3.2 Selection — `select` / `on_select` `[MVP]`
- [ ] BPP validates the selected item + slot against **live** availability — never against a possibly-stale cached/searched view (Source-of-Truth rule)
- **Test Gate:** `E2E`/`NEG` — selecting a slot that was booked by someone else microseconds earlier is correctly rejected, not silently accepted.

### 3.3 Initialization — `init` / `on_init` `[MVP]`
- [ ] Order draft preparation: real `Quotation` generation (`price`, `breakup[]`, `ttl` per the confirmed schema), fulfillment details, inventory revalidation
- **Test Gate:** `E2E` — `init` produces a real `Quotation` object matching the confirmed schema shape, not a placeholder.

### 3.4 Confirmation — `confirm` / `on_confirm` (the critical path) `[MVP]`
- [ ] Sequential, synchronous core: reserve (`HELD`) → confirm → inventory commit (no real payment step yet — deferred to the future `livetracker3.md`, per `project_details.md`'s payment KPI being explicitly out of this tracker's scope)
- [ ] Both [BAP_details_v1.1.md](BAP/BAP_details_v1.1.md) and [BPP_details_v1.1.md](BPP/BPP_details_v1.1.md) list a **Payment Module** as a named peer of Confirm/Cancel/etc. in their Transaction Module breakdown — so a minimal Payment Module interface exists here (real `Order.payments[]` field populated using the confirmed real `Payment.status` value, standardized-error-shaped `NOT_YET_IMPLEMENTED` response if actually invoked), rather than the module simply not existing. **Corrected against the real schema** (`beckn/protocol-specifications`, `schema/Payment.yaml`): `status` is a fixed 2-value enum, `PAID` / `NOT-PAID` — there is no `PENDING` value in the real protocol, so an earlier draft of this line citing one was fixed to use the real `NOT-PAID` value instead of inventing a state the schema doesn't have. This keeps the documented module list honestly represented without doing real gateway work now.
- [ ] Real `HELD` → `ACTIVE` (booking) transition with atomic capacity decrement
- [ ] Fires `BookingConfirmed` — triggers **parallel, async** downstream consumers (notification stub, analytics stub, cache invalidation) via `shared/event_bus`, per the sequential-core/parallel-aftermath rule
- **Test Gate:** `E2E`/`LOAD` — real confirm succeeds end-to-end; a concurrent-confirm race test (two customers, one capacity-1 slot) yields exactly one winner; the exact same confirm request retried produces one booking, not two (idempotency).

### 3.5 Post-Booking — `status` / `cancel` / `update` / `track` `[MVP]`
- [ ] `status`/`on_status` — read-only booking + fulfillment state lookup
- [ ] `cancel`/`on_cancel` — fires a `SlotReleased` compensation event (not a rollback), booking state → `CANCELLED`, inventory returned to `AVAILABLE`
- [ ] `update`/`on_update` — reschedule support (adjusts the booked slot, triggers inventory adjustment)
- [ ] `track`/`on_track` — minimal implementation returning the current fulfillment state (`SCHEDULED`/`IN_PROGRESS`/`COMPLETED`); genuine live-location tracking is not meaningful for a walk-in Beauty appointment and is correctly deferred to Phase 4.2's Automotive technician-dispatch case, but the endpoint itself exists now for protocol completeness — not silently dropped from the confirmed API surface
- **Test Gate:** `E2E` — all four flows tested live against a real prior booking, not stubbed.

### 3.6 Error Handling & Resilience for the Business Layer `[MVP]`
- [ ] **Two error shapes, deliberately, not one.** The real Beckn wire calls (BAP↔Gateway↔BPP: `search`/`select`/`init`/`confirm`/etc.) use the confirmed real protocol error shape (`type`/`code`/`message`/`path`) — that's what interoperability requires. The customer/business-facing web-to-backend calls (signup, login, the browser's "confirm" button, business dashboard calls) are explicitly **non-Beckn-protocol** surfaces per [API_CONVENTIONS.md](API_CONVENTIONS.md)'s own scope line ("BAP/BPP web-to-backend calls") and use that document's already-established shape (`{error: {code, message, field, correlation_id}}`) instead — never the protocol shape, and never a third ad-hoc shape invented for this tracker.
- [ ] Web-facing list endpoints (search results, a customer's booking history) use the established cursor-based pagination convention (`?cursor=...&limit=...`, per [API_CONVENTIONS.md](API_CONVENTIONS.md)) — not left unpaginated, and not offset-based
- [ ] New web-facing endpoints (signup, login, business dashboard APIs) live under `/api/v1/...` per the established versioning convention
- [ ] The browser-facing "confirm" trigger endpoint honors the established `Idempotency-Key` header convention ([API_CONVENTIONS.md](API_CONVENTIONS.md)) — a flaky mobile connection causing the browser to retry the confirm POST must not create two bookings, at the web layer, independent of and in addition to the event-level idempotency already covered in §1.4
- [ ] Retryable-vs-non-retryable error classification (timeouts/network = retryable; invalid data/unavailable slot = not)
- [ ] Reuse the existing `shared/resilient_http` circuit breaker (Redis-backed, proven in Phase 4.2 of `livetracker1.md`) for any new inter-service calls this phase introduces
- [ ] Custom user-facing error pages/states for the web UI (booking failed, session expired, slot no longer available) built on Phase 2.4's error-state component — never a raw stack trace or blank crash screen reaching a real user
- **Test Gate:** `NEG` — each of the 6 new flows has at least one deliberate negative-path test (invalid slot, expired hold, unauthorized caller) confirmed to fail cleanly with the standardized error shape and a comprehensible UI error state.

### 3.7 Business-Layer Security Hardening `[MVP]`
- [ ] Rate limiting on all new public-facing endpoints (`search`/`select`/`init`/`confirm`/signup/login) — these are now internet-exposed customer endpoints, a materially different threat profile from the trust layer's participant-only endpoints. **Redis-backed from day one, not in-memory.** **Precise, corrected framing:** this class of bug (per-worker in-memory state silently undercounting across gunicorn's multiple workers) has been hit once already in this project and fixed once, for real, live-verified (the circuit breaker, `RedisCircuitBreaker`, Phase 4.2 of `livetracker1.md`). Registry's own rate limiter (`rate_limit.py`) hit the *same* root cause but was explicitly **left unfixed and accepted** as an `[MVP]`/`[PILOT]`-stage limitation, deferred to `[BETA]` per `RUNBOOK.md` — it was never actually fixed, so citing it as a second "already fixed" precedent would be wrong. The lesson this task follows is the *proven fix* (the circuit breaker), while deliberately not repeating the *still-open, accepted-for-now* gap (the rate limiter) a second time in new code.
- [ ] Reservation-hold abuse prevention: cap concurrent `HELD` reservations per customer/IP, so a malicious actor can't deny service to real customers by mass-holding every slot without ever confirming (a direct DoS risk created by the TTL-hold mechanism itself, not a hypothetical)
- [ ] IDOR (Insecure Direct Object Reference) protection: every `status`/`cancel`/`update`/`track` call is authorized against the calling customer's own booking — Customer A can never view or modify Customer B's booking by guessing/incrementing an ID
- [ ] Input validation on every new customer-facing field (signup, search query, booking notes) — rejects malformed/oversized/injection-shaped input before it reaches business logic
- [ ] CSRF protection and secure session handling on the new web-facing auth forms
- [ ] Basic bot/scraping protection on `search` (rate-limit-based; not a full behavioral bot-detection system — that would be over-engineering at this stage)
- **Test Gate:** `SEC` — real abuse-simulation tests: rapid-fire reservation-hold spam is throttled; an authenticated customer attempting to fetch/cancel another customer's booking ID is rejected with `403`, not `404`-leaked or silently allowed; SQLi/XSS-shaped payloads against search and signup fields are rejected, not reflected or executed.

### 3.8 Performance, Latency & Cost Optimization `[MVP]`
- [ ] Read-through Redis cache in front of `search` — cache serves discovery reads; `confirm` always hits the authoritative Postgres data directly, never the cache (the exact split confirmed from the inventory-patterns research and the real Source-of-Truth principle already applied in §3.2). **Placement, explicit:** this is a new BAP/BPP-side catalog cache, separate infrastructure from Gateway's existing optional cache (`gateway-cache`, built in `livetracker1.md`) — Gateway's cache is documented as trust-layer only (participant identity/public keys, per [beckn_gateway_details_v1.1.md](beckn-gateway/beckn_gateway_details_v1.1.md)'s Data Persistence section), and stays untouched; the two must not be conflated into one cache serving both purposes.
- [ ] Write-through cache invalidation triggered by `SlotConfirmed`/`SlotReleased`/`SlotUpdated` events from `shared/event_bus` — a booked slot disappears from cached search results within one event-processing cycle, not on the next TTL expiry
- [ ] API latency measured for `search` (cached) vs. `confirm` (uncached, DB-authoritative) under local load — no fixed target number is set yet, honestly, since no real traffic baseline exists (same pattern already established in `RUNBOOK.md` for the trust layer); the measurement itself is the deliverable, not a guessed SLA
- [ ] Cost note: still local Docker only, zero new cloud spend — this task is about query/cache efficiency, not infrastructure sizing, which stays under Phase 0.9's cost-governance framework and only becomes relevant once real infrastructure is provisioned
