## A. Core Registry API â€” Confirmed from Actual OpenAPI Spec

Source (Tier A): [`api/registry/build/registry.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/api/registry/build/registry.yaml), [beckn/protocol-specifications](https://github.com/beckn/protocol-specifications)

1. **Three Registry paths exist in the core spec â€” no more, no fewer:**
   - `POST /subscribe` â€” request: `context` (action=`subscribe`) + `message` (a `Subscription` object + `nonce` UUID). Response: `{"status": "UNDER_SUBSCRIPTION"}`.
   - `POST /on_subscribe` â€” **registry-initiated callback into the participant**, not called by the participant. Request: `context` (action=`subscribe`) + `message.status` (see enum below) + `message.challenge` (string, required â€” "a random string generated by the registry, encrypted with the subscriber's encryption public key"). Response required from participant: `{"answer": "<decrypted challenge>"}`.
   - `POST /lookup` â€” request: a `Subscription` object used as a filter (any subset of fields). Response: **array of `Subscription` objects** matching the filter.
   - **There is no separate update, deactivate, or key-rotation endpoint.** Confirmed directly from the spec's path list. Rotation and status changes both happen by calling `/subscribe` again with the same `subscriber_id`.

2. **`Subscription` object fields (core schema):** `subscriber_id, url, type, domain, location, key_id, signing_public_key, encr_public_key, valid_from, valid_until, status, created, updated`. Note: the core spec names the encryption field `encr_public_key`; ONDC's live implementation uses `encryption_public_key` in practice (Â§B.5 below) â€” **use `encryption_public_key` when integrating against ONDC**, but expect naming drift if referencing the abstract spec directly.

3. **Status enum is core protocol, not an ONDC-only extension** (this corrects an earlier note in this document that had it mislabeled): `INITIATED, UNDER_SUBSCRIPTION, SUBSCRIBED, INVALID_SSL, UNSUBSCRIBED`, defined directly in `registry.yaml`. Reconciled lifecycle:
   - `INITIATED` â€” registrant whitelisted/approved by the Registrar (ONDC: Network Participant Portal approval), before any `/subscribe` call.
   - `UNDER_SUBSCRIPTION` â€” `/subscribe` call accepted by the registry; challenge pending.
   - `SUBSCRIBED` â€” `/on_subscribe` challenge answered correctly; participant is now a trusted "Subscriber."
   - `INVALID_SSL` / `UNSUBSCRIBED` â€” terminal/error states.

4. **Two key pairs per participant, not one:**
   - `signing_public_key` â€” Ed25519, used for request signing.
   - `encr_public_key` / `encryption_public_key` â€” X25519, used for the on_subscribe encrypted challenge.

5. **Registry acts as a Public Key Infrastructure (PKI).** Private keys never leave the participant; only public keys are ever submitted.

## B. Core Registry Onboarding Mechanics â€” ONDC-Specific Operational Layer

Your [project_details.md](project_details.md) lists "Beckn-ONDC Implementation Guidelines" as required reading. ONDC is a live, operated network running on top of the core spec above, and adds real operational mechanics the abstract spec doesn't cover. Source (Tier B): [ONDC onboarding doc](https://github.com/ONDC-Official/developer-docs/blob/main/registry/Onboarding%20of%20Participants.md), [example Subscribe payload](https://gist.github.com/Ajcode69/4f9064ceac713c4f42677b0d619fe3ba).

1. **Three separate registry deployments**, each independently whitelisted:

   | Environment | Subscribe Endpoint | Lookup Endpoint |
   |---|---|---|
   | Staging | `https://staging.registry.ondc.org/subscribe` | `https://staging.registry.ondc.org/v2.0/lookup` |
   | Pre-Production | `https://preprod.registry.ondc.org/ondc/subscribe` | `https://preprod.registry.ondc.org/v2.0/lookup` |
   | Production | `https://prod.registry.ondc.org/subscribe` | `https://prod.registry.ondc.org/v2.0/lookup` |

2. **Onboarding prerequisites beyond the core spec's `Subscription` object:**
   - Valid FQDN (becomes part of `subscriber_id`) and a valid SSL certificate (OCSP-validated by the registry).
   - **Portal whitelisting**: register on the ONDC Network Participant Portal, request environment access per environment; human-reviewed, typically 6â€“48 hours. This is the concrete form of "Registrar approval" that produces the `INITIATED` state in Â§A.3.
   - **Domain-ownership verification** (on top of the core on_subscribe challenge): sign a `request_id` with the signing private key (Ed25519, unhashed) and host it at `ondc-site-verification.html` on the participant's own domain; the registry fetches and validates it. This step doesn't appear in the abstract core spec at all â€” it's an ONDC operational addition, and it was missing from the original tracker.

3. **Real Subscribe payload is richer than the abstract `Subscription` object** â€” ONDC nests business/KYC verification fields inside it:

   ```json
   {
     "context": { "operation": { "ops_no": 2 } },
     "message": {
       "request_id": "uuid",
       "timestamp": "2024-04-17T06:08:00.000Z",
       "entity": {
         "gst": { "legal_entity_name": "...", "business_address": "...", "city_code": ["std:080"], "gst_no": "..." },
         "pan": { "name_as_per_pan": "...", "pan_no": "...", "date_of_incorporation": "DD/MM/YYYY" },
         "name_of_authorised_signatory": "...",
         "address_of_authorised_signatory": "...",
         "email_id": "...", "mobile_no": 9999999999, "country": "IND",
         "subscriber_id": "...", "unique_key_id": "uuid", "callback_url": "/on_subscribe_path",
         "key_pair": {
           "signing_public_key": "base64 Ed25519",
           "encryption_public_key": "base64 ASN.1 DER X25519",
           "valid_from": "2024-04-17T06:08:00.000Z",
           "valid_until": "2026-07-08T13:44:54.101Z"
         }
       },
       "network_participant": [
         { "subscriber_url": "https://.../ondc", "domain": "ONDC:RET10", "type": "sellerApp", "msn": true, "city_code": ["std:080"] }
       ]
     }
   }
   ```
   `ops_no`: `1` = BAP registration, `2` = BPP registration, `4` = both, `3`/`5` = **deprecated** (Seller-On-Record, obsolete â€” do not implement). `network_participant` is an array â€” one call can register a participant across multiple domains/roles.
   The GST/PAN/business-verification fields are ONDC's own compliance layer on top of the core `Subscription` object in Â§A.2 â€” expect them to be absent if targeting a non-ONDC Beckn network.

4. **Key rotation confirmed (combining Â§A.1's "no dedicated endpoint" with the schema's validity window):** `key_pair.valid_from`/`valid_until` means keys are not permanent; rotation is done by calling `/subscribe` again for the same `subscriber_id` with a new `key_pair`, before `valid_until` expires. There is no separate rotation endpoint â€” this is now settled, not a guess.

5. **On_subscribe challenge confirmed end-to-end:** registry POSTs `{"subscriber_id": "...", "challenge": "<encrypted>"}` to the participant's `callback_url`; challenge is encrypted with a shared key derived from the participant's encryption private key + the registry's public key; participant decrypts and responds `{"answer": "<decrypted>"}`; success â†’ ACK, failure â†’ NACK with an error code.

6. **Concrete rate limits:** Subscribe = 10 requests/minute; Lookup = 7,600 requests/minute.

7. **Concrete failure/NACK reasons** (use as negative test cases): subscriber_id not whitelisted (code 132), timestamp outside allowed window, domain-verification-file missing/invalid signature, duplicate subscriber_id, OCSP/SSL validation failure, challenge decryption failure, incorrect participant type, signature verification failure on the domain-verification file.

## C. Request Signing â€” Confirmed Full Header Syntax

Source (Tier B/C): [Subscriber Signing docs](https://developers.becknprotocol.io/docs/infrastructure-layer-specification/authentication/subscriber-signing/), [Gateway Signing docs](https://developers.becknprotocol.io/docs/infrastructure-layer-specification/authentication/gateway-signing/), [Signing Beckn APIs Draft03](https://sites.google.com/beckn.org/signing-beckn-apis-draft03/home)

1. **Algorithm:** Ed25519 (XEd25519 signature scheme). **Digest:** BLAKE-512 (BLAKE2b-512) of the request body, Base64-encoded. **Signing string:** `(created): {value}\n(expires): {value}\ndigest: BLAKE-512={digest}`.

2. **Full `Authorization` header syntax (participant-to-participant requests):**
   ```
   Authorization: Signature keyId="{subscriber_id}|{unique_key_id}|{algorithm}",algorithm="ed25519",created="{unix_ts}",expires="{unix_ts}",headers="(created) (expires) digest",signature="Base64(sign(signing_string))"
   ```
   `created`â†’`expires` delta should equal the request context's TTL; `expires` must not exceed the signing key's own expiry.

3. **Gateway uses a *different* header â€” this is a real protocol distinction, not interchangeable.** **Corrected (2026-07-19, see Â§H.3):** this line originally said `Proxy-Authorization`, sourced from the Tier-B/C docs listed above â€” those describe an earlier design that was itself corrected in the actual spec repository. The real, current header (per [`beckn/protocol-specifications` PR #149](https://github.com/beckn/protocol-specifications/pull/149), Tier A, merged into `core-0.9.3-draft`) is **`X-Gateway-Authorization`**, not `Proxy-Authorization` â€” the original name was deliberately replaced because `Proxy-Authorization` is a single-hop RFC 7235 header that intermediaries like load balancers consume and strip, breaking authentication across any real multi-hop deployment ([issue #119](https://github.com/beckn/protocol-specifications/issues/119)). The header VALUE format is unchanged (identical `Signature keyId=...` syntax to `Authorization`) â€” only the header NAME was wrong. Any HTTP client/middleware built for BAP/BPPâ†”Registry signing must not assume the same header name applies to Gateway-originated calls, and must use `X-Gateway-Authorization` specifically, not the deprecated `Proxy-Authorization`.

4. **Confirmed from the core spec (Â§A):** every path in both `registry.yaml` and `transaction.yaml` requires `SubscriberAuth` via a signed `Authorization` header â€” signing is not optional on any endpoint, including `/lookup`.

## E. Compliance & Certification Layer (distinct from API conformance â€” not yet reflected anywhere in the tracker before this)

Everything in Â§Aâ€“D is *technical protocol conformance*: does your code speak the right schema on the right endpoint. **Compliance/certification is a separate axis**: is your organization and deployment *allowed* to go live on the network at all. This project's own [project_details.md](project_details.md) references "Beckn-ONDC Implementation Guidelines," and Beckn's own governance model treats Certification as a first-class architectural layer alongside the Specification layer (Source, Tier B: [becknprotocol.io/governance](https://becknprotocol.io/governance/)) â€” so this is legitimately in scope, not scope creep.

1. **Pramaan â€” ONDC's mandatory certification test bench** ([pramaan.ondc.org](https://pramaan.ondc.org/), [ONDC-Official/pramaan](https://github.com/ONDC-Official/pramaan)). Four-stage gate before any participant can go live:
   - **Preparation** â€” register subscriber_id, select domain, review prerequisites.
   - **Testing** â€” automated test cases simulate real transaction flows against the participant's app, validating API responses and performance.
   - **Validation** â€” test reports reviewed, issues fixed, fixes re-verified.
   - **Certification** â€” complete mandatory flows, pass all required tests, obtain the Pramaan Integration Report â†’ cleared for production.
   Confirmed from source: a **Probationary Period on the Production environment** follows even after Pramaan certification, before full Go-Live is granted. Domain-specific test suites exist (Retail, Logistics, F&B) â€” relevant since this project targets healthcare/automotive/beauty, so expect to confirm which domain test suite(s) apply, or whether custom domains require a different certification path.
   **This is a real gate this tracker doesn't currently model** â€” Pramaan certification sits between Phase 3 (onboarding) and true production readiness in Phase 4, specifically for the *production* environment (staging/preprod onboarding in Phase 3 doesn't require full Pramaan sign-off, production does).

2. **ONDC Network Participant Agreement** ([resources.ondc.org/network-participant-agreement](https://resources.ondc.org/network-participant-agreement)) â€” a legal/contractual layer, not code: registering as a Network Participant is a formal agreement with ONDC (a Section 8 company), with binding obligations (anti-bribery, confidentiality, conflict-of-interest, non-discriminatory conduct). This is an organizational/legal action item, not an engineering task â€” flagged here so it isn't silently assumed to already be in place.

3. **IGM (Issue & Grievance Management) â€” mandatory, not optional**, per India's Consumer Protection (E-Commerce) Rules 2020. Confirmed technical shape:
   - `/report` and `/on_issue` API surface (initiator reports an issue, responder resolves it), with `context, issue_id, category_id, details, created_by, created_for, severity, state, created_at`.
   - Issue categories: `order, billing, fulfillment, operations, compliance, contract, settlement`. Severity: `critical, high, medium, low`. States: `initiated, registered, processing, escalated, resolved`.
   - **Every Network Participant must designate a Grievance Redressal Officer (GRO)** and share their details with ONDC â€” an organizational requirement, not a code requirement.
   IGM's actual API implementation is business-workflow-layer work (like search/select/confirm), correctly out of scope for *this* foundation/trust-layer tracker â€” but the **GRO designation and IGM-readiness awareness** belongs in the Phase 4 production-readiness review so it isn't discovered late.
   Source (Tier B/C): [ONDC IGM Explainer](https://ondc-static-website-media.s3.ap-south-1.amazonaws.com/ondc-website-media/downloads/governance-and-policies/a.ONDC's%20IGM-%20Explainer-%20v1.0.pdf), [ONDC IGM Policy](https://ondc-static-website-media.s3.ap-south-1.amazonaws.com/ondc-website-media/downloads/governance-and-policies/CHAPTER+%5B6%5D+Issue+and+Grievance+Management+Policy.pdf)

4. **DPDP Act, 2023 (India's data protection law) applies** â€” this system stores customer/booking personal data (per [project_details.md](project_details.md), a consumer-facing booking platform). Confirmed obligations: data fiduciaries need documented consent management, access/correction/grievance-redressal rights for data principals, and "reasonable security safeguards" scaled to data sensitivity. This is a real, non-optional compliance requirement for the eventual production system, distinct from â€” but reinforcing â€” the security-hardening tasks already in Phase 2.5/4.3.

5. **Beckn's own Certification governance layer** (not ONDC-specific): Beckn's architecture formally separates a "Specification and Support" layer from a "Certification" layer, with an accountable governance role over certification/compliance rules. Confirms compliance isn't an ONDC add-on bolted onto Beckn â€” it's a first-class concept in Beckn's own architecture, and any Beckn network (not just ONDC) would have some certification gate, even if the specific mechanics differ.

**Net effect on the tracker:** Phase 3 (participant onboarding) was previously modeled as if reaching `SUBSCRIBED` status were sufficient to transact on the network. It is not, for production: Pramaan certification, the Network Participant Agreement, and GRO/IGM-readiness are additional non-technical gates that sit between "trusted participant" (this tracker's actual scope) and "live on production" (a Phase 4 concern). ## Confirmed Against ONDC's Own Reference Implementation (2026-07-17)

Cross-checked `shared/beckn_crypto/crypto.py` against ONDC's actual reference toolkit â€”
`cryptic_utils.py` in [ONDC-Official/reference-implementations](https://github.com/ONDC-Official/reference-implementations/blob/main/utilities/signing_and_verification/python/cryptic_utils.py) â€”
closing the item at line 173 below (this document previously cited the wrong repo,
`developer-docs`; the actual reference code lives in `reference-implementations`).

- **BLAKE-512 digest, signing string format, Ed25519 sign/verify, `Authorization` header
  syntax â€” all confirmed to match exactly**, byte-for-byte construction. No changes needed.
- **Challenge encryption scheme â€” confirmed, and it did NOT match what this codebase had,
  fixed 2026-07-17.** ONDC's real scheme: the raw X25519 shared secret is used
  **directly** as an AES-256 key (no KDF/HKDF derivation step at all), with **AES-ECB
  mode** and PKCS7 padding. This codebase previously used HKDF-SHA256 â†’ AES-256-GCM â€” a
  more modern, more secure construction, but genuinely incompatible with the real
  network: a participant using the old scheme could not decrypt a real Registry's
  on_subscribe challenge, or vice versa. `shared/beckn_crypto/crypto.py` now matches
  ONDC's real scheme exactly (`encrypt_challenge`/`decrypt_challenge`/
  `_derive_shared_key`). ECB is a genuinely weaker mode than GCM (deterministic per
  block) â€” matched anyway because protocol-fidelity to the confirmed real scheme takes
  priority over a locally-preferred "more secure" alternative the network doesn't
  actually speak; documented as a real, known tradeoff in the module docstring, not
  silently upgraded back.
- **Private-key serialization is a real difference, but not an interop concern.** ONDC's
  `generate_key_pairs()` stores the Ed25519 signing private key as PyNaCl's internal
  64-byte `seed || pubkey` format and the X25519 encryption private key as PKCS8/DER;
  this codebase stores both as a raw 32-byte seed. This does NOT break interop â€”
  private keys are never transmitted between participants, only public keys (same
  32-byte raw / DER SubjectPublicKeyInfo format either way) and signatures/ciphertexts
  (now byte-compatible per the fix above). It only matters if someone tries to manually
  import a private key generated by one tool into the other â€” worth knowing, not worth
  changing our own consistent internal storage format for.

## H. Transaction Flow Mechanics â€” `search`/`on_search` (`livetracker2.md` Phase 3.1)

Researched before implementing Â§3.1, per this document's own rule (confirm against official sources before writing protocol-facing code, never guess). Three real, previously-open questions resolved:

1. **Async is mandatory, not a network policy option.** Source (Tier A): [`beckn/protocol-specifications` discussion #187](https://github.com/beckn/protocol-specifications/discussions/187) â€” a proposal to allow BPP APIs to be synchronous "as part of network policy" was explicitly considered and **rejected** by the spec maintainers: *"not in favor of networks deciding these as policies"*, *"all APIs should be designed and built for Async"*. Every `/action` call gets an immediate ACK/NACK in the same HTTP session; the real response arrives later as a separate signed `/on_action` callback call. **Direct consequence for this project:** Registry's existing `Subscribe`/`on_subscribe` challenge-response (synchronous, built in `livetracker1.md`) is a *different* mechanism â€” a key-ownership challenge-response handshake, already independently confirmed against the real ONDC onboarding sequence (Â§B) â€” not the same rule, not a deviation. `search`/`select`/`init`/`confirm`/etc. (this tracker's Phase 3) are the commerce transaction actions this async rule actually governs, and must be built accordingly.
2. **`Ack` schema.** Source (Tier A): [`schema/Ack.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/schema/Ack.yaml) â€” `status` (required, enum `ACK`/`NACK`), `tags` (optional, array of `TagGroup`). This is the literal body every `/action` and `/on_action` endpoint returns in its own immediate HTTP response, wrapped as `{"message": {"ack": {"status": "ACK"}}}` alongside the standard `context` echo.
3. **Gateway re-signs when forwarding â€” under a different header than some secondary docs still describe.** Source (Tier A, a real spec correction caught): the original design used a `Proxy-Authorization` header for Gateway's own outbound signature when forwarding BAP's request to a BPP. [`beckn/protocol-specifications` issue #119](https://github.com/beckn/protocol-specifications/issues/119) flagged this as wrong per RFC 7235 (`Proxy-Authorization` is a single-hop header, consumed and stripped by intermediaries like load balancers â€” it would never survive a real multi-hop deployment). [PR #149](https://github.com/beckn/protocol-specifications/pull/149) (merged into `core-0.9.3-draft`) replaced it with **`X-Gateway-Authorization`** â€” a custom header that survives intermediary hops. Some Tier-B/C sources (blog posts, docs pages that couldn't be fetched directly) still describe the old `Proxy-Authorization` name â€” **`X-Gateway-Authorization` is correct, per the actual merged spec change; the older name is stale and must not be used.** The original `Authorization` header (BAP's own signature) is still forwarded alongside it, not replaced â€” this specific detail is a reasonable inference from the multi-hop signing design (the BPP still needs to verify the original sender's identity, not just Gateway's), not a word-for-word quoted spec line; flagged honestly as inferred, not fully spec-quoted, same as this document's convention for the healthcare/automotive domain-code items below.
4. **`on_search` routes back through Gateway, not directly BPP â†’ BAP.** Source (Tier B, `developers.becknprotocol.io`'s Transaction Layer page, cross-checked against the Gateway's documented "lean, stateless routing server" role): *"The BAP will send a search request to a Beckn gateway, and the gateway will broadcast the search to different [BPPs]... The BPPs immediately call the `on_search` API which is sent back to the BG, and the BG then forwards those requests back to the BAPs."* Gateway is on the critical path for both directions of the `search`/`on_search` pair, not just the outbound leg â€” consistent with Gateway being the only party that did the Registry lookup for `SUBSCRIBED` participants and their callback URLs in the first place.

**Net effect on Phase 3.1 implementation:** Gateway's `/search` handler needs to (a) ACK the BAP caller immediately, (b) asynchronously forward the intent to each `SUBSCRIBED` Beauty BPP with a fresh `X-Gateway-Authorization` signature alongside the original `Authorization` header, (c) receive each BPP's async `/on_search` callback, and (d) forward that callback on to the BAP's own registered callback URL â€” Gateway is a real participant in the callback path, not a pass-through after the initial routing decision.

## I. Transaction Flow Mechanics â€” `select`/`on_select` (`livetracker2.md` Phase 3.2)

Researched before implementing Â§3.2, prompted by an audit of the initial design proposal that turned up a real, material correction â€” the proposal assumed `select`'s wire shape carried `item`/`fulfillment` directly, which is wrong.

1. **`/select` and `/on_select` both wrap everything in `message.order`, not a bare item/fulfillment.** Source (Tier A): [`api/transaction/build/transaction.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/api/transaction/build/transaction.yaml) â€” `/select` request: `message.order` (an `Order` object). `/on_select` response: `message.order` (+ optional `error`). The customer's selection is expressed as a real, if minimal, `Order` draft (`provider`, `items[]`, `fulfillments[]`) from the very first step, not a separate lighter-weight shape â€” the same `Order` object gets progressively enriched across select â†’ init â†’ confirm, per real Beckn convention.
2. **`on_select`'s returned `Order` includes a real `quote` (`Quotation`) â€” provisional pricing is expected at Selection, not deferred entirely to Init.** Source (Tier A): [`schema/Order.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/schema/Order.yaml) confirms `quote` as a top-level `Order` field (optional, but present in the real schema â€” `on_select` is where it's first populated per the spec's own request/response shape). `livetracker2.md`'s original Â§3.2 wording ("BPP validates the selected item + slot against live availability") didn't mention this â€” a real planning gap, corrected in Â§3.2 below.
3. **`Quotation.yaml`** â€” `id`, `price` (`Price` object), `breakup[]` (`item`/`title`/`price` per line), `ttl` (`Duration`). All optional per the schema; `breakup[]` is the full itemized detail Â§3.3 (Init) already correctly anticipates building in full â€” `on_select`'s quote can be the minimal `price`-only form without `breakup`, schema-honest without over-building an itemization step (fees/taxes/etc.) that has no real line items to itemize yet at Selection.
4. **`Price.yaml`** â€” `currency`, `value`, plus six optional situational variants (`estimated_value`/`computed_value`/`listed_value`/`offered_value`/`minimum_value`/`maximum_value`) for pricing scenarios (auctions, estimates) this project doesn't need â€” only `currency`/`value` are used here, the rest deliberately unused rather than populated with guessed values.
5. **`Time.yaml`** â€” `label`, `timestamp`, `duration`, `range` (start/end date-time), `days` (comma-separated weekday string), `schedule` (`$ref Schedule.yaml`). Confirms recurring time-of-day availability (e.g. "09:00, 10:00, 11:00, Monâ€“Sat") is expressible entirely through confirmed protocol fields, without needing to expose this project's own internal `Slot` primary keys on the wire â€” consistent with `Slot`'s own docstring already noting it "has no direct protocol-schema counterpart."

**Net effect on Phase 3.2 implementation, and two real gaps this research surfaced beyond the wire shape itself:**
- **No pricing exists anywhere in this codebase yet.** `Resource`/`Item` currently has no price field, and `build_beauty_catalog()` doesn't populate `Item.price` (Â§2.3's own "correctly omitted" list named `fulfillments`/`payments`/`offers` but never flagged price as intentionally missing â€” it was simply not built). A minimal `Resource.price` field (currency + value) is added in Â§3.2 to make `on_select`'s `quote` real rather than a schema-shaped placeholder with invented numbers â€” small, bounded, and legitimately in-scope now that the real schema confirms it's needed at this step, not scope creep.
- **BPP `SUBSCRIBED` status can go stale between search and select.** Gateway's `dispatch_search` re-checks `SUBSCRIBED` via a fresh Registry lookup on every call; a `select` forwarded straight to a remembered `bpp_uri` from an earlier search result without the same re-check would be a real trust-chain regression. Â§3.2 below adds this explicitly.

## K. Transaction Flow Mechanics â€” `confirm`/`on_confirm` (`livetracker2.md` Phase 3.4)

Researched before implementing Â§3.4. `/confirm`/`on_confirm`'s `message.order` wire shape was already confirmed in Â§J.1 during Â§3.3's research pass, so this section covers only the two facts genuinely new to Confirm.

1. **`Order.status`'s real enum, confirmed directly (Tier A, `schema/Order.yaml`)**: `ACTIVE` / `COMPLETE` / `CANCELLED` â€” "Allowed values can be defined by the network policy" per the schema's own description, but this project adopts the schema's own listed defaults verbatim rather than inventing separate ones. This exactly matches `shared/inventory_core/models.py`'s existing `Booking.Status` enum (`HELD`/`ACTIVE`/`COMPLETE`/`CANCELLED` â€” `HELD` being this project's own pre-order addition, already documented as such in that model's docstring) â€” a claim that model's docstring had been making since Phase 1.3 without an independent Tier-A citation recorded here; now closed.
2. **`Payment.yaml`'s full shape, confirmed directly (Tier A, `schema/Payment.yaml`)**: `id`, `collected_by`, `url`, `params`, `type` (enum: `PRE-ORDER`/`PRE-FULFILLMENT`/`ON-FULFILLMENT`/`POST-FULFILLMENT`), `status` (enum: `PAID`/`NOT-PAID` â€” already correctly cited in `livetracker2.md` Â§3.4's own text), `time`, `tags` â€” all optional. This project's minimal Payment Module placeholder populates only `status: "NOT-PAID"`; `type` is deliberately left unset (no real payment-stage decision has actually been made yet, so guessing one of the four enum values would be dishonest, not merely incomplete).

**Net effect on Â§3.4 implementation, and one real concurrency gap this research prompted, found by re-reading `confirm_hold()`'s existing Phase 1.3 implementation before reusing it, not after:** `confirm_hold()` read the target `Booking` with a plain unlocked `.get()`, then read-checked-then-wrote its status â€” under two genuinely concurrent `/confirm` calls for the *same* booking (a realistic double-submit/flaky-retry scenario, not a hypothetical), both could pass the "is this HELD" check before either committed, and each would independently fire a fresh `BookingConfirmed`/`SlotConfirmed` event â€” a real duplicate-notification risk, since Phase 1.4's event-idempotency layer dedupes by `event_id` (assigned per publish call), not by business key, so two independently-published "the same booking got confirmed" events would NOT be caught as duplicates downstream. Fixed by wrapping the read+check+transition in `transaction.atomic()` + `Booking.objects.select_for_update()`, and treating an already-`ACTIVE` booking as an idempotent no-op (return early, skip the event) rather than re-running the transition â€” `confirm_hold()`'s existing return contract (a bare `Booking`) is unchanged, so this is a non-breaking, backward-compatible fix.
