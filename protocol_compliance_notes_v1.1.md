## A. Core Registry API — Confirmed from Actual OpenAPI Spec

Source (Tier A): [`api/registry/build/registry.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/api/registry/build/registry.yaml), [beckn/protocol-specifications](https://github.com/beckn/protocol-specifications)

1. **Three Registry paths exist in the core spec — no more, no fewer:**
   - `POST /subscribe` — request: `context` (action=`subscribe`) + `message` (a `Subscription` object + `nonce` UUID). Response: `{"status": "UNDER_SUBSCRIPTION"}`.
   - `POST /on_subscribe` — **registry-initiated callback into the participant**, not called by the participant. Request: `context` (action=`subscribe`) + `message.status` (see enum below) + `message.challenge` (string, required — "a random string generated by the registry, encrypted with the subscriber's encryption public key"). Response required from participant: `{"answer": "<decrypted challenge>"}`.
   - `POST /lookup` — request: a `Subscription` object used as a filter (any subset of fields). Response: **array of `Subscription` objects** matching the filter.
   - **There is no separate update, deactivate, or key-rotation endpoint.** Confirmed directly from the spec's path list. Rotation and status changes both happen by calling `/subscribe` again with the same `subscriber_id`.

2. **`Subscription` object fields (core schema):** `subscriber_id, url, type, domain, location, key_id, signing_public_key, encr_public_key, valid_from, valid_until, status, created, updated`. Note: the core spec names the encryption field `encr_public_key`; ONDC's live implementation uses `encryption_public_key` in practice (§B.5 below) — **use `encryption_public_key` when integrating against ONDC**, but expect naming drift if referencing the abstract spec directly.

3. **Status enum is core protocol, not an ONDC-only extension** (this corrects an earlier note in this document that had it mislabeled): `INITIATED, UNDER_SUBSCRIPTION, SUBSCRIBED, INVALID_SSL, UNSUBSCRIBED`, defined directly in `registry.yaml`. Reconciled lifecycle:
   - `INITIATED` — registrant whitelisted/approved by the Registrar (ONDC: Network Participant Portal approval), before any `/subscribe` call.
   - `UNDER_SUBSCRIPTION` — `/subscribe` call accepted by the registry; challenge pending.
   - `SUBSCRIBED` — `/on_subscribe` challenge answered correctly; participant is now a trusted "Subscriber."
   - `INVALID_SSL` / `UNSUBSCRIBED` — terminal/error states.

4. **Two key pairs per participant, not one:**
   - `signing_public_key` — Ed25519, used for request signing.
   - `encr_public_key` / `encryption_public_key` — X25519, used for the on_subscribe encrypted challenge.

5. **Registry acts as a Public Key Infrastructure (PKI).** Private keys never leave the participant; only public keys are ever submitted.

## B. Core Registry Onboarding Mechanics — ONDC-Specific Operational Layer

Your [project_details.md](project_details.md) lists "Beckn-ONDC Implementation Guidelines" as required reading. ONDC is a live, operated network running on top of the core spec above, and adds real operational mechanics the abstract spec doesn't cover. Source (Tier B): [ONDC onboarding doc](https://github.com/ONDC-Official/developer-docs/blob/main/registry/Onboarding%20of%20Participants.md), [example Subscribe payload](https://gist.github.com/Ajcode69/4f9064ceac713c4f42677b0d619fe3ba).

1. **Three separate registry deployments**, each independently whitelisted:

   | Environment | Subscribe Endpoint | Lookup Endpoint |
   |---|---|---|
   | Staging | `https://staging.registry.ondc.org/subscribe` | `https://staging.registry.ondc.org/v2.0/lookup` |
   | Pre-Production | `https://preprod.registry.ondc.org/ondc/subscribe` | `https://preprod.registry.ondc.org/v2.0/lookup` |
   | Production | `https://prod.registry.ondc.org/subscribe` | `https://prod.registry.ondc.org/v2.0/lookup` |

2. **Onboarding prerequisites beyond the core spec's `Subscription` object:**
   - Valid FQDN (becomes part of `subscriber_id`) and a valid SSL certificate (OCSP-validated by the registry).
   - **Portal whitelisting**: register on the ONDC Network Participant Portal, request environment access per environment; human-reviewed, typically 6–48 hours. This is the concrete form of "Registrar approval" that produces the `INITIATED` state in §A.3.
   - **Domain-ownership verification** (on top of the core on_subscribe challenge): sign a `request_id` with the signing private key (Ed25519, unhashed) and host it at `ondc-site-verification.html` on the participant's own domain; the registry fetches and validates it. This step doesn't appear in the abstract core spec at all — it's an ONDC operational addition, and it was missing from the original tracker.

3. **Real Subscribe payload is richer than the abstract `Subscription` object** — ONDC nests business/KYC verification fields inside it:

   ```json
   {
     "context": { "operation": { "ops_no": 2 } },
     "message": {
       "request_id": "uuid",
       "timestamp": "2024-04-17T06:08:00.000Z",
       "entity": {
         "gst": { "legal_entity_name": "...", "business_address": "...", "city_code": ["std:080"], "gst_no": "..." },
         "pan": { "name_as_per_pan": "...", "pan_no": "...", "date_of_incorporation": "DD/MM/YYYY" },
         "name_of_authorised_signatory": "...",
         "address_of_authorised_signatory": "...",
         "email_id": "...", "mobile_no": 9999999999, "country": "IND",
         "subscriber_id": "...", "unique_key_id": "uuid", "callback_url": "/on_subscribe_path",
         "key_pair": {
           "signing_public_key": "base64 Ed25519",
           "encryption_public_key": "base64 ASN.1 DER X25519",
           "valid_from": "2024-04-17T06:08:00.000Z",
           "valid_until": "2026-07-08T13:44:54.101Z"
         }
       },
       "network_participant": [
         { "subscriber_url": "https://.../ondc", "domain": "ONDC:RET10", "type": "sellerApp", "msn": true, "city_code": ["std:080"] }
       ]
     }
   }
   ```
   `ops_no`: `1` = BAP registration, `2` = BPP registration, `4` = both, `3`/`5` = **deprecated** (Seller-On-Record, obsolete — do not implement). `network_participant` is an array — one call can register a participant across multiple domains/roles.
   The GST/PAN/business-verification fields are ONDC's own compliance layer on top of the core `Subscription` object in §A.2 — expect them to be absent if targeting a non-ONDC Beckn network.

4. **Key rotation confirmed (combining §A.1's "no dedicated endpoint" with the schema's validity window):** `key_pair.valid_from`/`valid_until` means keys are not permanent; rotation is done by calling `/subscribe` again for the same `subscriber_id` with a new `key_pair`, before `valid_until` expires. There is no separate rotation endpoint — this is now settled, not a guess.

5. **On_subscribe challenge confirmed end-to-end:** registry POSTs `{"subscriber_id": "...", "challenge": "<encrypted>"}` to the participant's `callback_url`; challenge is encrypted with a shared key derived from the participant's encryption private key + the registry's public key; participant decrypts and responds `{"answer": "<decrypted>"}`; success → ACK, failure → NACK with an error code.

6. **Concrete rate limits:** Subscribe = 10 requests/minute; Lookup = 7,600 requests/minute.

7. **Concrete failure/NACK reasons** (use as negative test cases): subscriber_id not whitelisted (code 132), timestamp outside allowed window, domain-verification-file missing/invalid signature, duplicate subscriber_id, OCSP/SSL validation failure, challenge decryption failure, incorrect participant type, signature verification failure on the domain-verification file.

## C. Request Signing — Confirmed Full Header Syntax

Source (Tier B/C): [Subscriber Signing docs](https://developers.becknprotocol.io/docs/infrastructure-layer-specification/authentication/subscriber-signing/), [Gateway Signing docs](https://developers.becknprotocol.io/docs/infrastructure-layer-specification/authentication/gateway-signing/), [Signing Beckn APIs Draft03](https://sites.google.com/beckn.org/signing-beckn-apis-draft03/home)

1. **Algorithm:** Ed25519 (XEd25519 signature scheme). **Digest:** BLAKE-512 (BLAKE2b-512) of the request body, Base64-encoded. **Signing string:** `(created): {value}\n(expires): {value}\ndigest: BLAKE-512={digest}`.

2. **Full `Authorization` header syntax (participant-to-participant requests):**
   ```
   Authorization: Signature keyId="{subscriber_id}|{unique_key_id}|{algorithm}",algorithm="ed25519",created="{unix_ts}",expires="{unix_ts}",headers="(created) (expires) digest",signature="Base64(sign(signing_string))"
   ```
   `created`→`expires` delta should equal the request context's TTL; `expires` must not exceed the signing key's own expiry.

3. **Gateway uses a *different* header — this is a real protocol distinction, not interchangeable.** **Corrected (2026-07-19, see §H.3):** this line originally said `Proxy-Authorization`, sourced from the Tier-B/C docs listed above — those describe an earlier design that was itself corrected in the actual spec repository. The real, current header (per [`beckn/protocol-specifications` PR #149](https://github.com/beckn/protocol-specifications/pull/149), Tier A, merged into `core-0.9.3-draft`) is **`X-Gateway-Authorization`**, not `Proxy-Authorization` — the original name was deliberately replaced because `Proxy-Authorization` is a single-hop RFC 7235 header that intermediaries like load balancers consume and strip, breaking authentication across any real multi-hop deployment ([issue #119](https://github.com/beckn/protocol-specifications/issues/119)). The header VALUE format is unchanged (identical `Signature keyId=...` syntax to `Authorization`) — only the header NAME was wrong. Any HTTP client/middleware built for BAP/BPP↔Registry signing must not assume the same header name applies to Gateway-originated calls, and must use `X-Gateway-Authorization` specifically, not the deprecated `Proxy-Authorization`.

4. **Confirmed from the core spec (§A):** every path in both `registry.yaml` and `transaction.yaml` requires `SubscriberAuth` via a signed `Authorization` header — signing is not optional on any endpoint, including `/lookup`.

## D. Core Transaction API (Gateway / BAP / BPP routing surface)

Source (Tier A): [`api/transaction/build/transaction.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/api/transaction/build/transaction.yaml)

Out of scope for this tracker's Phase 0–4 (business workflows are a future tracker per your explicit scoping), but the **endpoint contract shape** is directly relevant to the Gateway foundation work in Phase 1.2 and the trust-chain proof in Phase 4.1, so it's recorded here now rather than re-discovered later.

4. All responses follow the same ACK/NACK wrapper pattern seen in the Registry API.

## E. Compliance & Certification Layer (distinct from API conformance — not yet reflected anywhere in the tracker before this)

Everything in §A–D is *technical protocol conformance*: does your code speak the right schema on the right endpoint. **Compliance/certification is a separate axis**: is your organization and deployment *allowed* to go live on the network at all. This project's own [project_details.md](project_details.md) references "Beckn-ONDC Implementation Guidelines," and Beckn's own governance model treats Certification as a first-class architectural layer alongside the Specification layer (Source, Tier B: [becknprotocol.io/governance](https://becknprotocol.io/governance/)) — so this is legitimately in scope, not scope creep.

1. **Pramaan — ONDC's mandatory certification test bench** ([pramaan.ondc.org](https://pramaan.ondc.org/), [ONDC-Official/pramaan](https://github.com/ONDC-Official/pramaan)). Four-stage gate before any participant can go live:
   - **Preparation** — register subscriber_id, select domain, review prerequisites.
   - **Testing** — automated test cases simulate real transaction flows against the participant's app, validating API responses and performance.
   - **Validation** — test reports reviewed, issues fixed, fixes re-verified.
   - **Certification** — complete mandatory flows, pass all required tests, obtain the Pramaan Integration Report → cleared for production.
   Confirmed from source: a **Probationary Period on the Production environment** follows even after Pramaan certification, before full Go-Live is granted. Domain-specific test suites exist (Retail, Logistics, F&B) — relevant since this project targets healthcare/automotive/beauty, so expect to confirm which domain test suite(s) apply, or whether custom domains require a different certification path.
   **This is a real gate this tracker doesn't currently model** — Pramaan certification sits between Phase 3 (onboarding) and true production readiness in Phase 4, specifically for the *production* environment (staging/preprod onboarding in Phase 3 doesn't require full Pramaan sign-off, production does).

2. **ONDC Network Participant Agreement** ([resources.ondc.org/network-participant-agreement](https://resources.ondc.org/network-participant-agreement)) — a legal/contractual layer, not code: registering as a Network Participant is a formal agreement with ONDC (a Section 8 company), with binding obligations (anti-bribery, confidentiality, conflict-of-interest, non-discriminatory conduct). This is an organizational/legal action item, not an engineering task — flagged here so it isn't silently assumed to already be in place.

3. **IGM (Issue & Grievance Management) — mandatory, not optional**, per India's Consumer Protection (E-Commerce) Rules 2020. Confirmed technical shape:
   - `/report` and `/on_issue` API surface (initiator reports an issue, responder resolves it), with `context, issue_id, category_id, details, created_by, created_for, severity, state, created_at`.
   - Issue categories: `order, billing, fulfillment, operations, compliance, contract, settlement`. Severity: `critical, high, medium, low`. States: `initiated, registered, processing, escalated, resolved`.
   - **Every Network Participant must designate a Grievance Redressal Officer (GRO)** and share their details with ONDC — an organizational requirement, not a code requirement.
   IGM's actual API implementation is business-workflow-layer work (like search/select/confirm), correctly out of scope for *this* foundation/trust-layer tracker — but the **GRO designation and IGM-readiness awareness** belongs in the Phase 4 production-readiness review so it isn't discovered late.
   Source (Tier B/C): [ONDC IGM Explainer](https://ondc-static-website-media.s3.ap-south-1.amazonaws.com/ondc-website-media/downloads/governance-and-policies/a.ONDC's%20IGM-%20Explainer-%20v1.0.pdf), [ONDC IGM Policy](https://ondc-static-website-media.s3.ap-south-1.amazonaws.com/ondc-website-media/downloads/governance-and-policies/CHAPTER+%5B6%5D+Issue+and+Grievance+Management+Policy.pdf)

4. **DPDP Act, 2023 (India's data protection law) applies** — this system stores customer/booking personal data (per [project_details.md](project_details.md), a consumer-facing booking platform). Confirmed obligations: data fiduciaries need documented consent management, access/correction/grievance-redressal rights for data principals, and "reasonable security safeguards" scaled to data sensitivity. This is a real, non-optional compliance requirement for the eventual production system, distinct from — but reinforcing — the security-hardening tasks already in Phase 2.5/4.3.

5. **Beckn's own Certification governance layer** (not ONDC-specific): Beckn's architecture formally separates a "Specification and Support" layer from a "Certification" layer, with an accountable governance role over certification/compliance rules. Confirms compliance isn't an ONDC add-on bolted onto Beckn — it's a first-class concept in Beckn's own architecture, and any Beckn network (not just ONDC) would have some certification gate, even if the specific mechanics differ.

**Net effect on the tracker:** Phase 3 (participant onboarding) was previously modeled as if reaching `SUBSCRIBED` status were sufficient to transact on the network. It is not, for production: Pramaan certification, the Network Participant Agreement, and GRO/IGM-readiness are additional non-technical gates that sit between "trusted participant" (this tracker's actual scope) and "live on production" (a Phase 4 concern). Adding explicit tasks for these below.

## Confirmed Against ONDC's Own Reference Implementation (2026-07-17)

Cross-checked `shared/beckn_crypto/crypto.py` against ONDC's actual reference toolkit —
`cryptic_utils.py` in [ONDC-Official/reference-implementations](https://github.com/ONDC-Official/reference-implementations/blob/main/utilities/signing_and_verification/python/cryptic_utils.py) —
closing the item at line 173 below (this document previously cited the wrong repo,
`developer-docs`; the actual reference code lives in `reference-implementations`).

- **BLAKE-512 digest, signing string format, Ed25519 sign/verify, `Authorization` header
  syntax — all confirmed to match exactly**, byte-for-byte construction. No changes needed.
- **Challenge encryption scheme — confirmed, and it did NOT match what this codebase had,
  fixed 2026-07-17.** ONDC's real scheme: the raw X25519 shared secret is used
  **directly** as an AES-256 key (no KDF/HKDF derivation step at all), with **AES-ECB
  mode** and PKCS7 padding. This codebase previously used HKDF-SHA256 → AES-256-GCM — a
  more modern, more secure construction, but genuinely incompatible with the real
  network: a participant using the old scheme could not decrypt a real Registry's
  on_subscribe challenge, or vice versa. `shared/beckn_crypto/crypto.py` now matches
  ONDC's real scheme exactly (`encrypt_challenge`/`decrypt_challenge`/
  `_derive_shared_key`). ECB is a genuinely weaker mode than GCM (deterministic per
  block) — matched anyway because protocol-fidelity to the confirmed real scheme takes
  priority over a locally-preferred "more secure" alternative the network doesn't
  actually speak; documented as a real, known tradeoff in the module docstring, not
  silently upgraded back.
- **Private-key serialization is a real difference, but not an interop concern.** ONDC's
  `generate_key_pairs()` stores the Ed25519 signing private key as PyNaCl's internal
  64-byte `seed || pubkey` format and the X25519 encryption private key as PKCS8/DER;
  this codebase stores both as a raw 32-byte seed. This does NOT break interop —
  private keys are never transmitted between participants, only public keys (same
  32-byte raw / DER SubjectPublicKeyInfo format either way) and signatures/ciphertexts
  (now byte-compatible per the fix above). It only matters if someone tries to manually
  import a private key generated by one tool into the other — worth knowing, not worth
  changing our own consistent internal storage format for.

## F. Business Layer Schema Facts — `Time`/`Schedule`/`Descriptor`/`Item`/`Provider` (`livetracker2.md` Phase 1.1)

**Net effect on Phase 1.1:** `Resource` (the bookable person/thing) takes its descriptive fields from `Descriptor.yaml`'s real shape (`name`/`code`/`short_desc`/`long_desc`) plus `rateable`/`tags` from `Provider`/`Item`, rather than inventing arbitrary field names. `AvailabilityCalendar` takes `frequency`/`holidays`/`times` directly from the real `Schedule.yaml`, and the recurring-window/exception fields from `Time.yaml`'s `range`/`days`. `Slot` itself (a generated, concrete bookable window) has no direct protocol-schema counterpart — the real protocol only models availability at the `Time`/`Schedule` level and leaves per-slot generation to the provider's own inventory system, so `Slot`'s shape (`start_time`, `end_time`, `capacity_remaining`, `status`) is this project's own design, not a borrowed schema, consistent with `Fulfillment.state` already being confirmed project-defined territory.

## H. Transaction Flow Mechanics — `search`/`on_search` (`livetracker2.md` Phase 3.1)

Researched before implementing §3.1, per this document's own rule (confirm against official sources before writing protocol-facing code, never guess). Three real, previously-open questions resolved:

1. **Async is mandatory, not a network policy option.** Source (Tier A): [`beckn/protocol-specifications` discussion #187](https://github.com/beckn/protocol-specifications/discussions/187) — a proposal to allow BPP APIs to be synchronous "as part of network policy" was explicitly considered and **rejected** by the spec maintainers: *"not in favor of networks deciding these as policies"*, *"all APIs should be designed and built for Async"*. Every `/action` call gets an immediate ACK/NACK in the same HTTP session; the real response arrives later as a separate signed `/on_action` callback call. **Direct consequence for this project:** Registry's existing `Subscribe`/`on_subscribe` challenge-response (synchronous, built in `livetracker1.md`) is a *different* mechanism — a key-ownership challenge-response handshake, already independently confirmed against the real ONDC onboarding sequence (§B) — not the same rule, not a deviation. `search`/`select`/`init`/`confirm`/etc. (this tracker's Phase 3) are the commerce transaction actions this async rule actually governs, and must be built accordingly.
2. **`Ack` schema.** Source (Tier A): [`schema/Ack.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/schema/Ack.yaml) — `status` (required, enum `ACK`/`NACK`), `tags` (optional, array of `TagGroup`). This is the literal body every `/action` and `/on_action` endpoint returns in its own immediate HTTP response, wrapped as `{"message": {"ack": {"status": "ACK"}}}` alongside the standard `context` echo.
3. **Gateway re-signs when forwarding — under a different header than some secondary docs still describe.** Source (Tier A, a real spec correction caught): the original design used a `Proxy-Authorization` header for Gateway's own outbound signature when forwarding BAP's request to a BPP. [`beckn/protocol-specifications` issue #119](https://github.com/beckn/protocol-specifications/issues/119) flagged this as wrong per RFC 7235 (`Proxy-Authorization` is a single-hop header, consumed and stripped by intermediaries like load balancers — it would never survive a real multi-hop deployment). [PR #149](https://github.com/beckn/protocol-specifications/pull/149) (merged into `core-0.9.3-draft`) replaced it with **`X-Gateway-Authorization`** — a custom header that survives intermediary hops. Some Tier-B/C sources (blog posts, docs pages that couldn't be fetched directly) still describe the old `Proxy-Authorization` name — **`X-Gateway-Authorization` is correct, per the actual merged spec change; the older name is stale and must not be used.** The original `Authorization` header (BAP's own signature) is still forwarded alongside it, not replaced — this specific detail is a reasonable inference from the multi-hop signing design (the BPP still needs to verify the original sender's identity, not just Gateway's), not a word-for-word quoted spec line; flagged honestly as inferred, not fully spec-quoted, same as this document's convention for the healthcare/automotive domain-code items below.
4. **`on_search` routes back through Gateway, not directly BPP → BAP.** Source (Tier B, `developers.becknprotocol.io`'s Transaction Layer page, cross-checked against the Gateway's documented "lean, stateless routing server" role): *"The BAP will send a search request to a Beckn gateway, and the gateway will broadcast the search to different [BPPs]... The BPPs immediately call the `on_search` API which is sent back to the BG, and the BG then forwards those requests back to the BAPs."* Gateway is on the critical path for both directions of the `search`/`on_search` pair, not just the outbound leg — consistent with Gateway being the only party that did the Registry lookup for `SUBSCRIBED` participants and their callback URLs in the first place.

**Net effect on Phase 3.1 implementation:** Gateway's `/search` handler needs to (a) ACK the BAP caller immediately, (b) asynchronously forward the intent to each `SUBSCRIBED` Beauty BPP with a fresh `X-Gateway-Authorization` signature alongside the original `Authorization` header, (c) receive each BPP's async `/on_search` callback, and (d) forward that callback on to the BAP's own registered callback URL — Gateway is a real participant in the callback path, not a pass-through after the initial routing decision.

## I. Transaction Flow Mechanics — `select`/`on_select` (`livetracker2.md` Phase 3.2)

Researched before implementing §3.2, prompted by an audit of the initial design proposal that turned up a real, material correction — the proposal assumed `select`'s wire shape carried `item`/`fulfillment` directly, which is wrong.

1. **`/select` and `/on_select` both wrap everything in `message.order`, not a bare item/fulfillment.** Source (Tier A): [`api/transaction/build/transaction.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/api/transaction/build/transaction.yaml) — `/select` request: `message.order` (an `Order` object). `/on_select` response: `message.order` (+ optional `error`). The customer's selection is expressed as a real, if minimal, `Order` draft (`provider`, `items[]`, `fulfillments[]`) from the very first step, not a separate lighter-weight shape — the same `Order` object gets progressively enriched across select → init → confirm, per real Beckn convention.
2. **`on_select`'s returned `Order` includes a real `quote` (`Quotation`) — provisional pricing is expected at Selection, not deferred entirely to Init.** Source (Tier A): [`schema/Order.yaml`](https://raw.githubusercontent.com/beckn/protocol-specifications/master/schema/Order.yaml) confirms `quote` as a top-level `Order` field (optional, but present in the real schema — `on_select` is where it's first populated per the spec's own request/response shape). `livetracker2.md`'s original §3.2 wording ("BPP validates the selected item + slot against live availability") didn't mention this — a real planning gap, corrected in §3.2 below.
3. **`Quotation.yaml`** — `id`, `price` (`Price` object), `breakup[]` (`item`/`title`/`price` per line), `ttl` (`Duration`). All optional per the schema; `breakup[]` is the full itemized detail §3.3 (Init) already correctly anticipates building in full — `on_select`'s quote can be the minimal `price`-only form without `breakup`, schema-honest without over-building an itemization step (fees/taxes/etc.) that has no real line items to itemize yet at Selection.
4. **`Price.yaml`** — `currency`, `value`, plus six optional situational variants (`estimated_value`/`computed_value`/`listed_value`/`offered_value`/`minimum_value`/`maximum_value`) for pricing scenarios (auctions, estimates) this project doesn't need — only `currency`/`value` are used here, the rest deliberately unused rather than populated with guessed values.
5. **`Time.yaml`** — `label`, `timestamp`, `duration`, `range` (start/end date-time), `days` (comma-separated weekday string), `schedule` (`$ref Schedule.yaml`). Confirms recurring time-of-day availability (e.g. "09:00, 10:00, 11:00, Mon–Sat") is expressible entirely through confirmed protocol fields, without needing to expose this project's own internal `Slot` primary keys on the wire — consistent with `Slot`'s own docstring already noting it "has no direct protocol-schema counterpart."

**Net effect on Phase 3.2 implementation, and two real gaps this research surfaced beyond the wire shape itself:**
- **No pricing exists anywhere in this codebase yet.** `Resource`/`Item` currently has no price field, and `build_beauty_catalog()` doesn't populate `Item.price` (§2.3's own "correctly omitted" list named `fulfillments`/`payments`/`offers` but never flagged price as intentionally missing — it was simply not built). A minimal `Resource.price` field (currency + value) is added in §3.2 to make `on_select`'s `quote` real rather than a schema-shaped placeholder with invented numbers — small, bounded, and legitimately in-scope now that the real schema confirms it's needed at this step, not scope creep.
- **BPP `SUBSCRIBED` status can go stale between search and select.** Gateway's `dispatch_search` re-checks `SUBSCRIBED` via a fresh Registry lookup on every call; a `select` forwarded straight to a remembered `bpp_uri` from an earlier search result without the same re-check would be a real trust-chain regression. §3.2 below adds this explicitly.

## Remaining Open Items (genuinely unresolved, none blocking)

- Exact field-level `Order` schema fields not yet used (`billing`, `cancellation_terms`, `refund_terms`, `replacement_terms`, `return_terms`, `xinput`) — not yet needed for §3.2's minimal selection flow; will be confirmed before whichever later sub-phase (`init`/`confirm`/`cancel`) actually needs them, following the same just-in-time verification approach used throughout this document rather than front-loading research for phases not yet started.
- Whether ONDC's live registry deviates from the core `registry.yaml` in any field-naming edge case beyond the known `encr_public_key` vs. `encryption_public_key` drift (§A.2) — low risk, cheap to confirm on first real sandbox call.
- Beckn's emerging "DeDi" (Decentralized Directory) registry model in newer protocol versions is a **known future direction, not a current gap** — this project deliberately targets the classic centralized Registry pattern (matching [registry_details_v1.1.md](registry/registry_details_v1.1.md), the assignment brief, and ONDC's current live model). Revisit only if the target network migrates to DeDi.
- **UNVERIFIED and directly relevant to project scope, re-researched with a deeper pass — narrowed, not resolved:** exact ONDC `domain` codes for **healthcare** and **automotive** service-booking, as opposed to retail-of-goods.
  - **Confirmed:** `ONDC:RET13` = Beauty & Personal Care (retail) — still the only fully-confirmed code of the three target categories.
  - **Confirmed retail sub-codes exist in a documented pattern** (`RET10`=grocery, `RET11`=F&B, `RET12`=fashion, `RET13`=beauty & personal care, `RET14`=electronics, `RET15`=home & decor) — useful for pattern-recognition, but none of these are healthcare or automotive.
  - **Ruled out:** "Mobility" (`TRV` codes) — confirmed these cover ride-hailing/ticketing/travel booking, not automotive repair/service booking. Not the right category for this project's "automotive" service.
  - **New, better lead found but not resolved:** ONDC operates a real, distinct **"Services" vertical** — repo `ONDC-Official/ONDC-SRV-Specifications` and a live developer guide at `ondc-official.github.io/ONDC-SRV-Specifications/` exist, covering skilled/semi-skilled home services (the ONDC marketing site's own example: "fixing appliances to indulging in spa treatments at home"). This is a **much more plausible home for automotive repair/service-booking** than Mobility or Retail — but the actual `ONDC:SRV##` numeric code(s) and whether healthcare service-booking also lives under this vertical could not be extracted: the GitHub repo's tree only exposes a README (the real spec content is evidently built/deployed separately, not visible via the GitHub API), and the live developer-guide site's domain-code tables didn't render through available fetch tools.
  - **Net effect:** healthcare and automotive domain codes remain genuinely unconfirmed, but the search space has narrowed — the next verification step should be direct visual inspection of `https://ondc-official.github.io/ONDC-SRV-Specifications/` (its Swagger-UI/domain tables) by a human in a real browser, or contacting ONDC's Network Participant Portal support directly, rather than further automated search/fetch attempts, which have hit diminishing returns. **This affects the `domain` field value used in every Subscribe/Lookup/Search payload for two of the three target categories — confirm before Phase 3 onboarding for BPP domains beyond beauty.**
